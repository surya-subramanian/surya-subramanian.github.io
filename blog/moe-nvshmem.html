[Draft] MoE Training + Expert Parallelism

March 18, 2026

Mixture-of-experts training is useful because it increases parameter count without paying dense feed-forward compute for every token. Once experts are sharded across ranks, though, each MoE layer depends on routing, token packing, transport, expert execution, and restore. That path is where most of the systems work shows up.

A routed MoE layer has a straightforward high-level shape: score tokens with a router, choose a top-k set of experts, group tokens by destination rank, move them to the GPUs that own those experts, run the local expert MLPs, and then restore token order on the way back out.

Common bottlenecks

In practice, throughput usually degrades for one of three reasons:

  • too many tokens choose the same expert and create load imbalance,
  • packing and unpacking cost more than expected, or
  • the all-to-all exchange sits directly on the critical path.

The communication path matters most when expert parallelism spans multiple GPUs. If every rank has to build full send buffers, launch a collective, wait for the full receive side, and only then start expert compute, the sparse layer becomes a sequence of blocking stages.

Where NVSHMEM can help

NVSHMEM is useful here because it gives us a one-sided communication model across GPUs. Instead of expressing the whole dispatch as a single bulk exchange, we can think in terms of remote writes, readiness signals, and earlier consumption on the receiving rank. That maps fairly naturally onto MoE routing once expert destinations are known.

A sender can write token data into a remote staging region for a destination expert, signal that the slot is ready, and let the receiving rank start local work sooner. This does not solve routing skew or poor layouts, but it can reduce idle time between dispatch and expert execution.

Dispatch sketch

# High-level MoE flow
router_logits = gate(x)
topk_weight, topk_idx = torch.topk(router_logits, k=2, dim=-1)
topk_weight = torch.softmax(topk_weight, dim=-1)

packed_tokens, packed_meta = pack_by_destination(x, topk_idx)
remote_dispatch(packed_tokens, packed_meta)   # peer writes or collective exchange
expert_out = run_local_experts()
y = combine_and_restore(expert_out, packed_meta, topk_weight)

In a standard collective implementation, `remote_dispatch(...)` usually means building split sizes, running an all-to-all, and waiting for the receive side to be materialized before expert work begins. With an NVSHMEM-style approach, dispatch can be more incremental. 

// Simplified NVSHMEM-style pseudocode
for each token assigned to remote_expert:
    dst_rank = expert_to_rank[remote_expert]
    dst_slot = reserve_remote_slot(dst_rank, remote_expert)

    nvshmem_putmem_nbi(remote_buffer[dst_rank][dst_slot],
                       local_token_ptr,
                       token_bytes,
                       dst_rank)

    nvshmem_signal_op(remote_ready[dst_rank][dst_slot], 1, NVSHMEM_SIGNAL_SET, dst_rank)

// Receiver polls readiness and starts local expert work as slots arrive.

This is only a sketch, but it captures the pattern: write into expert-owned remote buffers, signal readiness, and give the receiver the option to start work before the entire exchange has finished. That is mainly useful when the local expert kernels are large enough to overlap with in-flight communication.

What still matters

Communication primitives are only one part of the picture. MoE layers still depend heavily on predictable capacity, low-overhead packing, contiguous layouts, and a router that does not collapse onto a small subset of experts. If any of those pieces are unstable, the communication path gets harder to hide no matter what API is underneath it.

In practice you still need:

  • stable capacity limits so experts do not overflow unpredictably,
  • layouts that keep packed token buffers contiguous and cheap to index,
  • enough local work to hide remote latency, and
  • a router that does not collapse onto a handful of experts.

Closing note

For expert parallel MoE, the dispatch path deserves as much attention as the expert kernel. NVSHMEM is interesting because it gives another way to structure that path with finer-grained data movement and synchronization, which can be a better fit for irregular token routing than a fully bulk-synchronous schedule.

Whether that tradeoff pays off depends on the layer shape, the interconnect, and how much overlap the implementation can actually sustain.
